# 풀림 기능 기획 Skill

> **목적**: 풀림 스터디의 개별 기능을 세부 기획할 때 일관된 원칙과 구조로 작업하기 위한 가이드  
> **적용 범위**: 풀림 스터디 14개 기능 전체, 향후 신규 기능 추가 시에도 동일 적용  
> **최종 업데이트**: 2026-04-15

---

## 1. 기획 원칙 (모든 기능 공통)

### 1.1 콘텐츠 1st 전략

> 기능적 우위보다 **콘텐츠가 만드는 해자**가 핵심이다.

- 모든 기능은 CUREA DEEP의 **3-Depth 교육과정 분류 체계**와 바인딩되어야 한다
- "범용 AI도 할 수 있는 것" vs "교육과정 데이터가 있어야만 할 수 있는 것"을 명확히 구분하고, 후자를 차별점으로 강조
- 기출 문항 DB, IRT 파라미터, 오답 패턴 DB 등 **데이터 자산**이 기능 경쟁력의 원천
- 3축(스튜디오-스토어-스터디) 연동 데이터 흐름을 반드시 설계에 포함

### 1.2 점진적 강화 모듈형 아키텍처

> 개별 기능은 독립적으로 동작하되, Core 기능 연결 시 자동으로 강화된다.

- **플러그인 슬롯 시스템**: 각 기능은 Core 기능(무한풀기, 튜터, 인덱스, 오답정복, 기억장치)과의 연결을 슬롯으로 정의
- **슬롯 상태 3가지**: `connected` (Core API 직접 호출), `fallback` (자체 LLM으로 기본 기능), `disabled` (비활성)
- **도메인 어댑터**: 새 도메인(중등, 자격증, 기업교육) 추가 시 어댑터 JSON 1개로 코드 변경 0줄

### 1.3 3-Tier AI 파이프라인

> 단순히 비싼 모델을 쓰는 게 아니라, 요청 유형에 따라 최적 모델을 라우팅한다.

| Tier | 모델 | 용도 | 비용 |
|------|------|------|------|
| T1: Edge | 룰엔진 + 임베딩 | 분류, 검증, 필터링 | ~₩0 |
| T2: Fast | Haiku / GPT-4o-mini | 단순 생성, Q&A, 알림 | ₩5~15/회 |
| T3: Deep | Sonnet / GPT-4o | 복잡 생성, 분석, 리포트 | ₩50~200/회 |

- 모든 기능의 각 요청에 대해 **어떤 Tier를 사용하는지** 명시
- 비실시간 작업(리포트, 분석)은 **Batch API**(50% 할인)로 처리
- **Prompt Caching**, **Semantic Cache**, **Structured Output** 기법 적극 활용

### 1.4 역할(Role) 기반 설계

> 학원·학교·학부모·과외를 동시에 지원하는 역할 체계

| 역할 | 코드 | 설명 |
|------|------|------|
| 관리자 | owner | 기관 소유자 (원장, 교감) |
| 운영자 | manager | 교사, 강사, 과외선생 |
| 보조자 | assistant | 보조교사, 학부모 |
| 학습자 | learner | 학생 |

각 기능의 모든 화면/기능에 대해 역할별 접근 권한을 명시.

### 1.5 도메인 확장성

> 고등 내신에서 시작하되, 중등/대학/자격증/기업까지 동일 아키텍처로 확장 가능해야 한다.

- 도메인 어댑터에 포함할 항목:
  ```json
  {
    "domain_id": "string",
    "curriculum_tree": "교육과정 3-Depth JSON 경로",
    "content_sources": ["콘텐츠 소스 목록"],
    "slot_config": { "기능별 슬롯 상태" },
    "role_presets": { "역할 프리셋" },
    "naming": { "도메인별 브랜드명 변형" },
    "pricing_tier": "과금 등급"
  }
  ```
- 새 도메인 추가 시 필요 작업: 교육과정 분류 JSON + 임베딩 계산 + 어댑터 JSON

## 3. 기획 프로세스

### 3.1 브레인스토밍 단계

1. **프로젝트 컨텍스트 확인**: 마스터 문서(01~04) + 기존 세부 기획서 읽기
2. **5가지 핵심 질문**:
   - 콘텐츠 소스: 어떤 교육과정 데이터/문항 DB를 쓰는가?
   - 타겟 고객: 학원/학교/학부모/과외 중 1차 타겟은?
   - 실행 환경: 풀림 앱 내 임베디드 vs 외부 연동?
   - AI 파이프라인: 어떤 모델을 어떻게 쓰는가?
   - 네이밍: 내부 코드명 + 외부 브랜드명 방향은?
3. **접근법 제안**: 최소 2~3개 설계 접근법 + 트레이드오프 + 추천

### 3.2 설계 단계

설계를 4개 섹션으로 나눠서 순서대로 승인:

1. **공유 아키텍처**: 플러그인 슬롯 + AI 파이프라인 + 역할 시스템
2. **기능 A 상세**: 모듈 구조, 핵심 기능, 데이터 흐름
3. **기능 B 상세**: (있을 경우)
4. **네이밍 + 시장 분석 + 사용자 여정**: 외부 브랜드 후보 3개, 시장 타당성, 시나리오 3+개

### 3.3 문서화 단계

1. 승인된 설계를 `풀림_마스터문서/` 디렉토리에 MD 파일로 작성
2. 파일명 규칙: `{번호}_{풀림}_{브랜드명}_{세부기획}.md`
3. 작성 후 spec self-review 수행
4. 사용자 최종 검토

## 4. 네이밍 규칙

### 4.1 이중 네이밍 체계

| 구분 | 용도 | 규칙 | 예시 |
|------|------|------|------|
| 내부 코드명 | 개발·문서·코드 | 영문 PascalCase, 기능 본질 | ClassBot, Planner |
| 외부 브랜드명 | 마케팅·UI·학생 대면 | "풀림 + 직관적 한국어 1~2단어" | 풀림 수업방, 풀림 시간표 |

### 4.2 브랜드명 선정 기준

1. **3초 이해**: 이름만 보고 무슨 기능인지 즉시 파악
2. **행동 기반**: 기술 명칭이 아니라 사용자 행동을 반영
3. **풀림 패밀리**: "풀림 OO" 형태 유지
4. **도메인 적응**: 고등용/성인용 등 도메인별 변형 가능해야 함

### 4.3 브랜드명 제안 시

- 최소 3개 후보 제시
- 각 후보의 장점/단점/적합 도메인 명시
- 추천안 + 추천 이유 포함
- 도메인별 네이밍 매트릭스 포함

## 5. 비판적 검토 기준

기획 시 아래 항목을 반드시 점검:

### 5.1 콘텐츠 의존도 체크

- [ ] "이 기능은 3-Depth 분류 없이도 동작하는가?" → 동작하면 차별화가 약한 것
- [ ] "범용 AI(ChatGPT)가 바로 복제할 수 있는가?" → 복제 가능하면 해자가 없는 것
- [ ] "데이터가 쌓일수록 기능이 좋아지는가?" → 플라이휠이 없으면 구조적 문제

### 5.2 비용 효율성 체크

- [ ] 모든 요청이 T3(고비용 모델)를 타지 않는가?
- [ ] 전체 요청의 60% 이상이 T1+T2로 처리 가능한가?
- [ ] 비실시간 작업은 Batch API로 처리하고 있는가?
- [ ] 사용자당 월 AI 비용이 구독료의 30% 이하인가?

### 5.3 확장성 체크

- [ ] 새 도메인 추가 시 코드 변경이 0줄인가?
- [ ] 도메인 어댑터 JSON 1개로 전환 가능한가?
- [ ] 슬롯 미연결 상태에서도 기본 가치를 제공하는가?

### 5.4 시장 타당성 체크

- [ ] 대한민국 교육 시장에서 실제 수요가 있는가?
- [ ] 초기 타겟(학원? 학교? 학부모?)이 명확한가?
- [ ] B2B/B2G 도입 절차 시간을 고려한 진입 전략이 있는가?
- [ ] 가격 민감도를 고려한 과금 구조인가?
