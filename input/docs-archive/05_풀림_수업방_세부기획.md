## 1. 한줄 정의

> **"교사가 만들고 통제하는 AI 학습방. 학생은 배우고, 교사는 모든 것을 본다."**

---

## 2. 왜 이 기능이 필요한가

### 2.1 시장 문제

| 문제 | 현상 | 규모 |
|------|------|------|
| ChatGPT 무단 사용 | 학생이 숙제·과제에 AI를 통제 없이 사용, 교사가 모니터링 불가 | 고등학생 73%가 AI 사용 경험 (2025 교육부 조사) |
| 답 베끼기 | AI가 답을 바로 알려줘서 사고 과정 생략 | 교사 82%가 AI 사용 부작용 우려 |
| 학습 효과 미측정 | 학생이 AI와 뭘 했는지 교사가 알 수 없음 | 기존 AI 도구의 리포팅 기능 부재 |
| 범위 이탈 | 수업 주제와 무관한 질문에 AI가 응답 | 범용 AI의 구조적 한계 |

### 2.2 풀림 수업방의 답

범용 AI와의 결정적 차이는 **"교육과정 분류 체계(3-Depth)에 바인딩된 AI"**라는 점.

| 범용 AI가 하는 것 | 수업방만 하는 것 |
|----------------|----------------|
| 문제 풀이 설명 | "이 문제는 수학I > 삼각함수 > 사인법칙 응용이고, **네가 이 유형에서 7번 중 5번 틀렸어**" |
| 힌트 제공 | "이 문제의 출제 의도는 '코사인→사인법칙 전환 판단'이야. **3번 선지가 함정인 이유**부터 생각해봐" |
| 범위 제한 | "오늘 수업 범위는 수학I-삼각함수-사인법칙이고, **이 범위의 기출 변형 3문제**를 내줄게" |
| 학습 리포트 | "학생A는 '조건부 확률' 개념은 이해했지만, **문제에서 사건 독립을 전제로 깔 때 놓치는 패턴**이 있습니다" |

이것은 3-Depth 교육과정 분류 + IRT 파라미터 + 오답 패턴 DB가 결합된 CUREA DEEP이 없으면 구현 불가능.

### 2.3 경쟁사 비교

| 역량 | Google Classroom | 클래스팅 AI | Khanmigo | ChatGPT Edu | **풀림 수업방** |
|------|-----------------|-----------|----------|-------------|--------------|
| 교사 AI 범위 통제 | - | 기본 | 제한적 | 기본 | **5단계 정밀 통제** |
| 실시간 모니터링 | - | - | 기본 | - | **풀 대시보드** |
| 이탈 감지·경고 | - | - | - | - | **AI 자동 감지** |
| 사고 유도 모드 | - | - | - | - | **5단계 스캐폴딩** |
| 학습 품질 자동 채점 | - | - | 기본 | - | **5영역 종합 평가** |
| 리포트 자동 생성 | 기본 | 기본 | 기본 | - | **6종 상세 리포트** |
| 한국 교육과정 최적화 | - | 부분 | - | - | **3-Depth 분류 기반** |

## 3. 서비스 대상

### 3.1 역할(Role) 기반 사용자 체계

```
┌──────────────────────────────────────────────┐
│                  역할 체계                     │
├──────────┬───────────────────────────────────┤
│ 관리자    │ owner    │ 기관 소유자 (원장, 교감)  │
│ 운영자    │ manager  │ 교사, 강사, 과외선생      │
│ 보조자    │ assistant│ 보조교사, 학부모          │
│ 학습자    │ learner  │ 학생                     │
└──────────┴───────────────────────────────────┘
```

### 3.2 타겟별 프리셋

| 프리셋 | 역할 매핑 | 기본 보안 | 기본 응답 모드 | 핵심 니즈 |
|--------|----------|----------|--------------|----------|
| **학원** | owner=원장, manager=강사 | Level 2 (단원) | 사고유도 | 수업 보조 + 숙제 관리 |
| **학교** | owner=관리자, manager=교사 | Level 2 (단원) | 사고유도 | 통제된 AI 수업 + 리포트 |
| **과외** | owner+manager=과외선생 | Level 1 (잠금) | 사고유도 | 1:1 맞춤 수업 보조 |
| **자녀학습** | owner+manager=학부모 | Level 3 (교과) | 가이드 | 자녀 학습 관리 + 모니터링 |

## 4. 모듈 구조

### 4.1 전체 아키텍처

```
classbot/
├── core/
│   ├── session-manager/       # 세션 생성·종료·상태 관리
│   ├── message-router/        # 3-Tier 모델 라우터 (T1→T2→T3)
│   ├── scope-guard/           # 5단계 보안 범위 제어
│   └── conversation-engine/   # 대화 상태 머신 + 컨텍스트 관리
│
├── intelligence/
│   ├── drift-detector/        # 이탈 감지 (T1: 룰+임베딩)
│   ├── thinking-scaffold/     # 사고 유도 5단계 (튜터 슬롯 연동)
│   ├── response-modes/        # 4가지 응답 모드 전환기
│   └── quality-scorer/        # 학습 품질 자동 평가 (T2)
│
├── control-panel/             # 운영자(교사) 제어 인터페이스
│   ├── bot-builder/           # 노코드 봇 설정 위저드
│   ├── dashboard/             # 실시간 모니터링
│   ├── report-generator/      # 6종 리포트 (T3: Batch)
│   └── intervention/          # 실시간 개입 도구
│
├── adapters/
│   ├── curriculum-adapter/    # 도메인별 교육과정 바인딩
│   └── content-adapter/       # 콘텐츠 소스 연결 (스토어/스튜디오)
│
└── slots/                     # 플러그인 슬롯 인터페이스
    ├── infinity-slot/         # 무한풀기 연동
    ├── tutor-slot/            # 튜터 연동
    ├── conqueror-slot/        # 오답정복 연동
    ├── memory-slot/           # 기억장치 연동
    ├── visual-slot/           # 비주얼 연동 (개념 설명 시)
    └── exam-slot/             # 모의고사 연동 (시험 모드)
```

> **참고**: 슬롯 체계는 풀림 스터디 전체에서 공통이며, 6개 글로벌 슬롯(infinity, tutor, conqueror, memory, visual, exam) 중 각 기능이 필요한 슬롯만 활성화합니다. 수업방은 주로 infinity·tutor·conqueror·memory 4개를 핵심으로 사용하고, visual·exam은 보조적으로 사용합니다.

### 4.2 플러그인 슬롯 시스템

Core 기능이 있으면 연결, 없으면 자체 폴백으로 독립 동작:

| 슬롯 | connected 동작 | fallback 동작 |
|------|---------------|--------------|
| infinity-slot | IRT 기반 문제 출제 (무한풀기 API) | T3로 범위 내 문제 생성 (실시간) |
| tutor-slot | 5단계 소크라테스 튜터 세션 | T2로 교육과정 프롬프트 기반 힌트 |
| conqueror-slot | 패턴 기반 오답 정복 세트 | 최근 오답 재출제 (T2) |
| memory-slot | 스페이스드 리피티션 세션 | 플래시카드 자체 생성 (T2) |
| visual-slot | VLM 인터랙티브 모듈 | 개념 카드 + 텍스트 설명 (T2) |
| exam-slot | 범위 지정 모의고사 (시험 모드) | T3로 시험 문제 세트 생성 |

**슬롯 상태 판정**: 도메인 어댑터의 `slot_config`에서 정의. 과목별로도 다를 수 있음. 슬롯 스키마는 `00_풀림_기능기획_Skill.md`의 표준 어댑터 패턴을 따름.
```json

  "english": { "infinity": "connected", "tutor": "connected", "conqueror": "connected", "memory": "connected", "visual": "fallback", "exam": "connected" },
  "korean": { "infinity": "fallback", "tutor": "fallback", "conqueror": "fallback", "memory": "connected", "visual": "fallback", "exam": "fallback" }
```

## 5. Scope Guard — 5단계 보안 범위

### 5.1 레벨 정의

| Level | 이름 | 범위 정의 | 작동 방식 | 적합 상황 |
|-------|------|----------|----------|----------|
| 1 | **잠금** | 업로드 자료 + 지정 주제만 | 교육과정 3-Depth 노드 화이트리스트 | 시험 직전 특정 단원 집중 |
| 2 | **단원** | 설정된 과목·단원 범위 내 | 3-Depth 중단원까지 허용 | 일반 수업 보조 |
| 3 | **교과** | 해당 과목 전체 | 대단원 레벨 전체 허용 | 자기주도 학습 시간 |
| 4 | **교육** | 모든 교과 + 학습 관련 주제 | 교육 카테고리 화이트리스트 | 프로젝트/리서치 |
| 5 | **개방** | 전체 (유해 필터 상시) | 블랙리스트 방식 (유해만 차단) | 창의 탐구, 진로 |

### 5.2 범위 체크 기술 구현

```
학생 메시지 → 임베딩 생성 (사전 계산 모델)
  → 교육과정 3-Depth 노드와 코사인 유사도 계산
  → 가장 가까운 노드의 depth가 허용 범위 내?
    → YES: 통과 → T2/T3로 응답 생성
    → NO: "이 주제는 오늘 수업 범위 밖이야. [허용범위] 관련 질문을 해볼까?"
    → 경계선 (유사도 0.6~0.75): T2로 한번 더 판정
```

**비용 포인트**: 범위 체크는 **T1 (임베딩 유사도)** — LLM 호출 없음, 비용 ~₩0.  
교육과정 노드 임베딩은 사전 계산 후 Pinecone/Redis에 캐싱.

## 6. 이탈 감지 시스템

### 6.1 감지 유형별 구현

| 이탈 유형 | 감지 방법 (Tier) | 대응 | 교사 알림 |
|----------|----------------|------|----------|
| **주제 이탈** | T1: Scope Guard 임베딩 유사도 < 0.6 | 자동 리다이렉트 | 누적 3회 시 |
| **답 베끼기 시도** | T1: 패턴 매칭 ("답 알려줘", "정답이 뭐야") + T2: 의도 분류 | 사고유도 모드 강제 전환 | 즉시 |
| **부적절 질문** | T1: 유해 키워드 필터 + T2: 맥락 분류 | 즉시 차단 | 즉시 |
| **장시간 무활동** | T1: 타이머 (기본 5분, 설정 가능) | 넛지 메시지 전송 | 10분 초과 시 |
| **반복 우회 시도** | T1: 동일 의도 3회 이상 카운트 | 세션 일시 정지 | 즉시 |
| **무의미 입력** | T1: 길이/엔트로피 체크 + T2 보조 | 경고 메시지 | 3회 누적 시 |

### 6.2 대응 강도 설정 (교사 조절 가능)

| 강도 | 이탈 시 동작 | 적합한 상황 |
|------|------------|-----------|
| **관대** | 안내 메시지만. 차단 없음. 교사에게 로그 | 자유 탐구 시간 |
| **보통** (기본) | 안내 → 3회 반복 시 주제 리다이렉트 → 교사 알림 | 일반 수업 |
| **엄격** | 즉시 리다이렉트 + 교사 즉시 알림 | 시험 모드, 과제 제출 |

## 7. 4가지 응답 모드

### 7.1 모드별 상세 동작

#### 모드 1: 사고유도 (기본) — 튜터 슬롯 연동

```
학생: "이 문제 모르겠어요"
봇(L1): "어떤 부분에서 막혔는지 말해줄 수 있어?" (방향 제시)
학생: "식을 세우는 게 어려워요"
봇(L2): "이 문제에서 구하려는 값이 뭐야?" (핵심 파악)
학생: "삼각형의 넓이요"
봇(L3): "삼각형 넓이 공식 중에 두 변과 끼인각을 알 때 쓰는 건?" (구체 단서)
학생: "S = 1/2 ab sinC요?"
봇(L4): "맞아! 그러면 이 문제에서 a, b, C에 해당하는 값은?" (거의 정답)
→ 정답 시: "잘했어!" + [infinity-slot connected] 유사 문제 1개 추천
→ 5단계까지 실패 시: L5에서 해설 공개
```

#### 모드 2: 가이드
- 힌트와 방향을 적극 제시
- 요청 시 단계별 설명 제공
- 답 자체는 학생이 입력해야 확인

#### 모드 3: 직접답변
- 범위 내 질문에 즉시 답변
- 풀이 과정 포함
- 효율 중시 상황 (복습, 빠른 개념 확인)

#### 모드 4: 시험
- 문제 출제만 수행
- 힌트·답변 일절 차단
- 제출 후 채점 + 해설 제공
- [infinity-slot 연동] IRT 기반 난이도 조절 문제 출제

### 7.2 시간대별 모드 스케줄링

교사가 수업 흐름에 맞춰 모드를 시간대별로 사전 설정 가능:

```
09:00-09:30  시험 모드 (쪽지시험)
09:30-10:00  사고유도 모드 (풀이 복습)
10:00-10:20  가이드 모드 (자유 질문)
```

## 8. 봇 빌더 (노코드 설정 위저드)

### 8.1 6단계 설정 흐름

```
Step 1: 기본 정보
  → 봇 이름 (기본값: "[과목] 수업방"), 과목, 대상 학년

Step 2: 범위 설정
  → 교육과정 3-Depth 트리에서 체크박스로 선택
  → 또는 "고1 수학 2단원" 자연어 입력 → 자동 매핑

Step 3: 참고자료 업로드 (선택)
  → PDF/이미지 업로드 → 자동 OCR + 임베딩
  → 스토어 교재 연결 (구매한 교재 자동 목록)

Step 4: 응답 모드 설정
  → 기본 모드 선택 + 시간대별 스케줄 (선택)

Step 5: 보안 레벨
  → 5단계 중 선택 + 이탈 시 대응 강도 설정

Step 6: 학생 초대
  → 코드/링크/QR 생성 → 학생이 스캔하여 참여
```

### 8.2 프리셋 템플릿

| 템플릿 | 보안 | 응답 모드 | 주요 설정 |
|--------|------|----------|----------|
| 내신 대비반 | Level 1 잠금 | 사고유도 + 시험 스케줄 | 특정 단원만, 시험 모드 포함 |
| 자기주도 학습실 | Level 3 교과 | 가이드 | 과목 전체 범위, 자유 질문 |
| 과외 세션 | Level 2 단원 | 사고유도 | 학부모 리포트 활성화 |
| 프로젝트 수업 | Level 4 교육 | 직접답변 | 교과 외 주제 허용 |
| 숙제방 | Level 1 잠금 | 사고유도 | 숙제 범위만, 답 요청 차단 |

## 9. 리포트 시스템

### 9.1 6종 리포트

| 리포트 | 대상 | 생성 시점 | AI Tier | 핵심 내용 |
|--------|------|----------|---------|----------|
| **실시간 대시보드** | 교사 | 상시 | T1 | 참여현황, 이탈감지, 활성상태 |
| **수업 종료** | 교사 | 세션 종료 시 | T2 Batch | 참여율, 주요 질문, 이탈 이력 |
| **학생 개인** | 교사/학부모 | 주 1회 | T3 Batch | 사고 깊이 평가, 성장 추이, 취약점 |
| **기간별 추이** | 교사 | 월 1회 | T3 Batch | 반 전체 성적 변화, 콘텐츠 효과 |
| **학부모** | 학부모 | 주 1회 | T2 Batch | 학습량, 참여도, 성장 요약 (비전문가용) |
| **기관** | 원장/관리자 | 월 1회 | T3 Batch | 강사별 활용도, ROI |

### 9.2 학습 품질 자동 평가 기준

| 영역 | 비중 | 측정 방법 |
|------|------|----------|
| 참여도 | 20% | 질문 횟수, 응답 시간, 세션 지속 시간 |
| 사고 깊이 | 30% | AI 힌트 사용 패턴, 자기 설명 길이/품질(T2 평가) |
| 과제 달성도 | 30% | 정답률, 완료율 |
| 자기주도성 | 10% | 추가 질문, 심화 탐구, 자발적 복습 |
| 협력 기여도 | 10% | (그룹 모드 시) 동료 질문 답변, 토론 참여 |

### 9.3 비용 최적화

- 실시간 대시보드: **T1 (₩0)** — 소켓 이벤트 집계만
- 수업 종료/학부모 리포트: **T2 Batch API (50% 할인)** — 세션 종료 후 비동기
- 학생 개인/기간별/기관 리포트: **T3 Batch API** — 야간 배치 처리

## 10. AI 파이프라인

### 10.1 3-Tier 모델 라우터

| Tier | 모델 | 용도 | 호출당 비용 (추정) |
|------|------|------|-----------------|
| **T1: Edge** | 룰엔진 + 임베딩 유사도 | 이탈 감지, 범위 체크, 의도 분류, 슬롯 라우팅 | ~₩0 |
| **T2: Fast** | Claude Haiku / GPT-4o-mini | 힌트 생성, 단순 Q&A, 동기부여, 리포트(간단) | ₩5~15 |
| **T3: Deep** | Claude Sonnet / GPT-4o | 문제 생성(CUREA DEEP), 패턴 분석, 상세 리포트 | ₩50~200 |

### 10.2 요청 흐름

```
학생 메시지 입력
  → T1: Scope Guard (임베딩 유사도, <10ms, ₩0)
    → 범위 외: 리다이렉트 메시지 (T1에서 종료)
    → 범위 내: 계속
  → T1: 이탈 감지 (패턴 매칭 + 룰엔진, <5ms, ₩0)
    → 이탈: 대응 처리 (T1에서 종료)
    → 정상: 계속
  → T1: 의도 분류 (임베딩 매칭, <10ms, ₩0)
    → 단순 개념 질문 → T2 응답 (₩5~15)
    → 풀이 과정 분석 → T3 + CUREA 컨텍스트 (₩50~200)
    → 격려/동기부여 → T2 템플릿 + 경량 개인화 (₩3~5)
    → 문제 요청 → 슬롯 라우팅 (infinity-slot API 또는 T3 생성)
```

### 10.3 비용 절감 기법

| 기법 | 설명 | 절감 효과 |
|------|------|---------|
| **Prompt Caching** | 교육과정 시스템 프롬프트 캐싱 (Anthropic/OpenAI) | 15~25% |
| **Semantic Cache** | 동일/유사 질문 임베딩 캐시 (Redis + 벡터 DB) | 10~20% |
| **Batch Processing** | 리포트·분석 등 비실시간 작업은 Batch API (50% 할인) | 5~10% |
| **Structured Output** | JSON 스키마 강제로 파싱 실패 재호출 방지 | 3~5% |
| **스트리밍 중단** | 이탈 감지 시 LLM 스트리밍 즉시 중단, 토큰 절약 | 2~5% |

### 10.4 예상 비용 (학원 30명반 기준)

| 항목 | 월 예상 사용량 | 단가 | 월 비용 |
|------|-------------|------|--------|
| T1 (범위체크+이탈감지) | 30명 × 50회/일 × 22일 = 33,000회 | ₩0 | ₩0 |
| T2 (힌트+Q&A) | 33,000회 × 50% = 16,500회 | ₩10 | ₩165,000 |
| T3 (문제생성+분석) | 33,000회 × 10% = 3,300회 | ₩100 | ₩330,000 |
| T3 Batch (리포트) | 30명 × 4주 × 3종 = 360건 | ₩150 | ₩54,000 |
| **합계** | | | **₩549,000/월** |

판매가 ₩500,000~1,000,000/월 → **마진 양호**

## 12. 데이터 흐름 & 3축 연동

### 12.1 내부 데이터 흐름

```
[학생 대화 로그]
  → 오답 데이터 → conqueror-slot → 패턴 DB 축적
  → 개념 질문 빈도 → 인덱스 → 취약 단원 자동 갱신
  → 정답/오답 이력 → memory-slot → 복습 스케줄 반영
  → 풀이 과정 텍스트 → 풀이분석 (향후) → 메타인지 리포트
```

### 12.2 3축 연동

| 방향 | 데이터 | 활용 |
|------|--------|------|
| 수업방 → 스토어 | 교재 연결 데이터 | "이 교재로 학습한 학생 N명" 표시 |
| 수업방 → 스토어 | 수업 효과 데이터 | 교재 효과 검증 리뷰에 기여 |
| 스토어 → 수업방 | 구매 교재 목록 | 봇 빌더 Step 3에서 자동 표시 |
| 수업방 → 스튜디오 | 콘텐츠 부족 신호 | "이 단원 문제가 부족합니다" 생산 우선순위 |
| 수업방 → 플래너 | 숙제 출제 데이터 | 학생 시간표에 숙제 블록 자동 삽입 |
| 플래너 → 수업방 | θ 업데이트 | 클래스봇 AI 응답 난이도 자동 조정 |

```
[Day 0] 강사 김쌤, 수업방 첫 사용

[Day 1] 수업 중
  → "지금부터 20분간 수업방에서 사인법칙 문제 풀어보세요"
  → 학생A: "사인법칙이 뭐예요?" → 봇: "삼각형에서 변과 각의 관계를 떠올려봐"
  → 학생B: "답 알려줘" → 이탈 감지 → 사고유도 전환 → 김쌤 알림
  → 학생C: 5분 무활동 → 넛지 메시지 자동 전송
  → 대시보드: 8명 활성, 이탈 1건, 평균 질문 3.2건

[수업 후]
  → 자동 수업 종료 리포트 생성
  → 학생별 사고 깊이 점수: A(72), B(45), C(68)

[Week 2]
  → 주간 리포트: "학생B의 답 요청 빈도 감소 (7회→2회)"
  → 학부모 리포트 자동 발송
```

### 13.2 학부모

```
[Day 0] 학부모 박씨, 고1 자녀 학습 관리
  → "자녀학습" 프리셋 선택
  → 수학I + 영어, 보안 Level 3, 가이드 모드
  → 자녀에게 초대 코드 전달

[매일]
  → 자녀가 수업방에서 자기주도 학습
  → 모르는 개념 → AI에게 질문 → 가이드 모드로 설명 + 힌트

[주 1회]
  → 학부모 리포트 수신
  → "이번 주 학습 세션 5회, 수학 정답률 72%→78%, 영어 독해 취약"
```

### 13.3 과외 선생

```
[수업 전]
  → 오늘 수업용 수업방 생성: 수열 등비급수, Level 1 잠금
  → 자체 프린트 PDF 업로드 → OCR → 봇이 해당 문제 기반으로 응답
  → 모드 스케줄: 30분 사고유도 → 20분 가이드

[수업 중]
  → 학생이 수업방에서 질문, AI가 힌트 제공
  → 과외 선생은 대면 보충 설명
  → AI가 풀이 과정 실시간 기록

[수업 후]
  → 세션 리포트: "등비급수 합 공식 적용 3회 오류"
  → 오답정복 슬롯에 해당 패턴 자동 등록
```
