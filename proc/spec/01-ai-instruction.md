# 01. AI 명령지침 (Core Rules)

이 명세를 읽고 개발할 AI(Claude, Codex 등)에게 전달할 행동 규칙. 풀림 프로젝트의 모든 작업에 반드시 적용된다.

---

## 1. 절대 원칙

### 1.1 깊게 사고하고 스스로 검토하라
- 추측 대신 명세·코드·문서를 확인하라
- 의심스러운 결정은 사용자에게 먼저 묻고 진행하라

### 1.2 사용자 요구사항 의도를 충실히 구현하라
- 요청된 화면·기능만 만든다
- "개선", "리팩터", "추가 페이지" 같은 자발적 제안을 먼저 하지 않는다
- 명세에 없는 기능은 임의로 추가하지 말고 사용자에게 확인한다

### 1.3 한국어로 작성하라
- 코드 주석·UI 카피·문서 모두 한국어
- 영문 코드명(Infinity Engine, Socrates, Conqueror 등)은 **내부 라우트·컴포넌트 이름에만** 사용. 학생 UI에는 **절대 노출하지 않는다**

### 1.4 콘텐츠 1st 전략 — 3-Depth 바인딩
- 모든 학습 콘텐츠(문제·개념·처방)는 CUREA DEEP의 3-Depth 교육과정 분류 체계와 연결된 상태로 표시
- 과목 → 단원 → 성취 기준 경로를 breadcrumb 또는 메타로 표기 가능해야 함
- "범용 AI가 할 수 있는 것"이 아닌 **"교육과정 데이터가 있어야만 할 수 있는 것"**을 UI에서 드러내라 (출제 의도, IRT 난이도, 오답 패턴 등)

---

## 2. 타겟 사용자 — 고등학생 (1순위 고정)

**고1~고3, 재수생** (수능·내신 준비자). 모든 화면 결정은 이 사용자를 기준으로 한다.

- 2차(중학생 자기주도 학습자)·3차(학원·학부모) 관점은 보조적으로만 고려
- 학생 뷰가 기본, **교사/학부모/관리자 뷰는 별도 라우트로 분리** (`/teacher/*`, `/parent/*`)
- UI 카피는 반말 아닌 친근한 존댓말 또는 중립 톤. "~해요", "~볼까요?"
- 정보 밀도: 학생이 시험기간에도 빠르게 파악 가능해야 함 (한 화면 = 한 의사결정)
- **모바일 우선** (학생 주 디바이스). 데스크탑은 교사·학부모 뷰에 우선

---

## 3. 3-Tier AI 파이프라인 인식

각 AI 호출 지점에서 어떤 Tier를 쓰는지 명시 (주석 또는 UI):

| Tier | 모델 | 용도 | 응답 시간 | UI 패턴 | 비용/요청 |
|------|------|------|----------|--------|---------|
| **T1 Edge** | 룰엔진+임베딩 | 분류·검증·필터링 | 즉시 | 로딩 UI 불필요 | ~₩0 |
| **T2 Fast** | Haiku/GPT-4o-mini | 단순 생성·Q&A·알림 | 1~3초 | 스피너 정도 | ₩5~30 |
| **T3 Deep** | Sonnet/GPT-4o | 복잡 생성·분석·리포트 | 5초+ | 스트리밍 또는 작업 큐 | ₩50~300 |

### 비용 효율 기준 (절대)
- 전체 AI 요청의 **60% 이상**이 T1+T2(저비용)로 처리 가능해야 함
- 비실시간 작업은 **Batch API(50% 할인)** 처리
- 사용자당 월 AI 비용 ≤ 구독료의 30%
- Prompt Caching + Semantic Cache + Structured Output 적극 활용

### 스트리밍 강제
- 스트리밍 가능한 지점에는 항상 **실시간 토큰 스트리밍 UI** 제공 (튜터·코치·리포트)

---

## 4. 브랜드 네이밍 (절대 원칙)

- 모든 기능은 **"풀림 ○○"** 형태 유지 (UI 카피·네비게이션·페이지 제목 전부)
- 학생이 기능명을 **3초 안에 이해**할 수 있어야 함
- 영문 코드명은 학생 UI 비노출 (앞 1.3 참조)

### 브랜드명 ↔ 영문 코드 매핑은 [03-features-and-ia.md](03-features-and-ia.md) 라우트표 참조

---

## 5. 플라이휠 가시화

학습 행동이 다음 루프로 연결되는 지점을 **학생에게도** 보여준다:

- "이 문제 10개 풀면 → 네 IRT θ가 업데이트돼요"
- "오답 정복 완료 → 기억장치 복습 큐에 자동 추가됨"
- 숨기지 말고 살짝 드러내는 것이 전환 비용(lock-in)의 핵심

---

## 6. 기능 간 연동 표시

스터디 14개 기능은 **서로 연결**되어야 의미가 있음. 화면마다:
- "이 결과를 **풀림 오답정복**으로 보내기" 같은 크로스 링크
- 풀림 코치(Agent Orchestra)는 모든 기능의 상위 오케스트레이터 역할 — 대시보드·홈에 항상 접근 가능

---

## 7. 고등학생 UX 디테일

- **시험 D-day 요소**를 전체 shell에 노출 (플래너가 활성화된 경우)
- **집중 모드** / **타이머**는 공통 shell 레벨에서 제공 (무한풀기·모의고사·플래너 실행 중)
- 학습 중단 없는 흐름 우선: 모달보다 사이드 패널/인라인
- 데이터 자산 표현: 히트맵(단원별), 레이더 차트(사고유형), 타임라인(기억장치 망각 곡선) — Recharts 기반

### 락인 선언 (사용자가 다음과 같이 선언하면 그 도메인 편집 한정)
> "풀림 Q 락인 작업" / "풀림 플래너 락인으로 진행" / "지금부터 클래스봇만 손질"

### 락인 중 자유롭게 해야 하는 것 (read · orchestration)
- 다른 도메인의 페이지/컴포넌트/mock **read** — 일관성 확인
- 공유 영역(`components/shell/*`, `lib/mock/{features,domains,...}.ts`) **read** — 라우트·메타 정합 확인
- `proc/spec/*` **read** — 도메인 권위 문서 참조

### 락인 중 하면 안 되는 것 (사용자 명시 확인 필요)
- 다른 도메인의 페이지/컴포넌트/mock **edit** → 그 도메인 락인 작업으로 별도 분리
- 공유 코드 **edit** → "글로벌 작업"으로 컨텍스트 전환 후 진행
- `lib/mock/{features,domains}.ts` **edit** → 메타 변경은 도메인 boundary를 흔들 수 있음

## 9. 신규 화면 구성 워크플로우 (필수)

새 화면을 만들라는 요청을 받으면 **반드시 이 순서**:

1. **명세 확인**: `proc/spec/03-features-and-ia.md` 해당 기능 섹션 + `proc/spec/04-ux-flow.md`의 시나리오를 Read
2. **학생이 경험하는 흐름**을 화면 플로우의 뼈대로 채택
3. **기술적 해자** ([05-business-rules.md](05-business-rules.md))에서 UI에 드러낼 요소 추출 (예: IRT 난이도 표시, 사고유도 모드 배지)
4. **타 기능 연동 지점** 모두 식별 (3축 연동 매트릭스 표 참조)
5. Mock 데이터 정의 (실 데이터 없이도 시나리오가 돌아가야 함)
6. 반응형 — 모바일 우선, 데스크탑은 2단 레이아웃
7. 완성 후 **플라이휠 가시화 체크**: 이 화면이 만드는 데이터가 어디로 흐르는지 1문장 주석

### 완성 기준
- 학생이 해당 기능의 **핵심 가치**(`03-features-and-ia.md` 한 줄 설명)를 3초 안에 이해하는가
- 타 기능으로의 **크로스 링크**가 최소 1개 있는가
- AI 호출 지점에 **Tier 표시**가 있는가 (주석 또는 UI)

---

## 10. 하지 말 것 (Anti-patterns)

- ❌ 영문 코드명(Infinity Engine, Socrates 등)을 학생 UI에 노출
- ❌ 범용 AI(ChatGPT스러운) UI 패턴 — 풀림은 "교육과정 기반 AI"라는 차별점이 드러나야 함
- ❌ `input/design-prototype/`의 기존 프로토타입 코드를 **그대로 복사**. 토큰·패턴만 참고하고 새로 작성
- ❌ 교사/학부모 기능을 학생 화면에 혼재. 반드시 라우트 분리
- ❌ 명세에 없는 기능 임의 추가
- ❌ 사용자가 요청하지 않은 "개선", "리팩터", "추가 페이지" 제안 먼저 하기
- ❌ 락인 도메인 안에서 다른 도메인 또는 공유 코드 편집

## 11. 검증 명령

작업 완료 시 정합성 확인:
```bash
cd web
pnpm exec tsc --noEmit  # 타입 체크
pnpm build              # 라우트 prerender 확인
pnpm lint               # ESLint 통과
```

라이브 검증은 `pnpm dev` (포트 3030) 또는 `docker compose up`.

## 12. 배포 정책

- **검증 명령 통과 후에만** 배포·머지 (lint·build·typecheck 모두 0 errors)
- 라우트 변경 시 **redirect 매핑** 필수, 옛 라우트로의 직링 0건 확인 후 폴더 삭제 가능
- 도메인 boundary를 변경하는 PR은 사용자 명시 승인 필요
