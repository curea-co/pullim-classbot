# 04. 사용자 경험 설계 (UX Flow / Use Case)

## 1. 학생 핵심 흐름 (6단계 루프)

```
[진단] 풀림 인덱스(/q/analysis/diagnose) — 15문 IRT 진단으로 학습 체력 측정
   ↓
[처방] 취약점 기반 학습 경로 추천 — 약점 단원·필요 블록 자동 제시
   ↓
[풀이] 풀림 무한풀기(/q/infinity/solve) — 적응형 문제, 무한 생성
   ↓
[해설] 풀림 해설 12-섹션 + 풀림 튜터 5단계 질문 가이드
   ↓
[정복] 풀림 복습 정복(/q/review/conquer) — 오답 패턴 분석 → Leitner 5-Box 자동 배정
   ↓
[갱신] 풀림 복습 전체(/q/review) — 개인별 망각 곡선 기반 복습 제안
   ↓ (루프 반복)
```

**완성 기준 (마스터 문서)**: 진단 → 처방 → 풀이 → 해설 → 정복 → 갱신 8단계 click-through에서 dead-end 0건.

## 2. Use Case (학생 시나리오)

### UC-1: 오답 정복 완전 사이클

```
1. 모의고사 50분 풀이
   → 오답 2개 발생

2. 채점·분석 (즉시)
   → θ 0.30 → 0.35로 상향
   → 오답 클러스터링 ("공식 대입 오류", "반전어 오독")

3. 풀림 해설 각 오답별 열람 (6~10분)
   → Section 1~12 순차 탐색
   → Section 3 4-Path 중 RECOMMENDED 경로 집중
   → Section 5 Error Anatomy: 내 실수 vs 정답 비교
   → Section 7 Visual Canvas (인터랙티브)
   → Section 9 Feynman Challenge (2분 셀프 설명)
   → Section 12 Memory Anchor 카드 저장

4. 오답 정복 (Leitner BOX 1)
   → 내일 복습 자동 큐에 추가

5. 플래너 연동
   → "2주 취약 보강" 블록 자동 생성
   → 취약 단원 시간 배분 증가

6. 일주일 후
   → 같은 패턴 재도전 → 3회 연속 성공 → BOX 2로 진급
```

### UC-2: 시간표 → Q 풀이 자동 진입

```
1. 플래너 홈, NEXT BLOCK: "수학 삼각함수 8문"

2. [지금 시작하기] 클릭
   → /q/infinity/solve?block=b3 딥링크

3. 상단 컨텍스트 바 표시
   - 과목: 수학 · 미적분 / 도함수 응용
   - 출처: 📅 플래너
   - 진행률: 0/8 (0%)

4. 연습 모드 (기본)
   - 좌: 첫 문제 렌더
   - 우: AI 코치 패널 (5단계 힌트, Scope L3 기본)

5. 8문 모두 풀이
   - 오답 2개 → 자동 Leitner BOX 1 배정

6. 블록 완료 → 플래너로 복귀
   - XP +80 부여
   - 스트릭 🔥 연장

7. 저녁 8시: 오답 정복 큐 알림
   - "어제 틀린 패턴 2개, 지금 복습할래요?"
```

### UC-3: 모의고사 급속 분석·처방

```
1. 모의고사 세션 선택 ("2025 6월 모평 수학")

2. 100분 실전 (OMR 마킹, 중간 AI 감독관 기록)

5. 액션 제안 (레몬 배경)
   - "경계조건 집중 훈련" → 오답정복 set 추천

6. 플래너 확인
   - 내일부터 "도함수 경계조건·적분 범위" 블록 자동 삽입
```

## 5. Navigation Flow (페이지 이동 경로)

### 5.1 플래너 → Q → 복습 순환

```
플래너 홈
  ↓
NEXT BLOCK: "수학 8문"
  ↓ [지금 시작하기]
Q 무한풀기 (solve)
  ├─ 실행률: 0/8 (0%)
  ├─ 또는 시험 모드 전환
  └─ AI 코치 패널 우측
  ↓ 블록 완료
플래너로 복귀
  ↓
저녁 BOX 복습 노티
Q 복습 (review)
  ├─ /q/review/wrong (Leitner 5-Box)
  │  └─ 오답 2개 BOX 1 재도전
  └─ /q/review (전체 망각곡선)
     └─ 일주일 전 오답·배운 것 due 복습
  ↓ 3회 연속 성공 → BOX 2로 진급
완료 → Q 홈 복귀 또는 다음 블록
```

### 5.2 클래스봇 실시간 수업 → Q 진입

```
클래스봇 실시간 수업 화면
  ↓
학생: "이 문제 모르겠어요"
  ↓
봇이 Scope L3 응답 (개념만)
  ↓
학생: "더 깊이" 버튼 또는 [Q에서 풀어보기]
  ↓
Q 무한풀기 (Scope L5, 복습 모드)
  ├─ 같은 개념 유사 문제 3개
  └─ 풀림 해설 전체 노출
  ↓
완료 후 클래스봇으로 복귀 또는 플래너로
```

### 5.3 풀림 코치(Agent Orchestra) — 모든 기능의 상위 오케스트레이터

```
어디서든 [코치에게] 버튼 또는 모바일 FAB
  ↓
/q/talk?tab=coach&context=<현재 컨텍스트>
  ↓
6 에이전트 중 적합한 입구 추천
  ├─ 학습 매니저 — 진도 관리
  ├─ 문제해결사 — 풀이 막힘 해소
  ├─ 오답박사 — 패턴 분석
  ├─ 시험전략가 — 모의고사 준비
  ├─ 멘탈코치 — 동기 부여
  └─ 플래너 — 일정 조정
```

## 6. 핵심 인터랙션 패턴

### 6.1 AI 코치 5단계 힌트 (튜터 모드)

```
문제 보다가 [힌트 요청 💬]
  ↓ T2 실행 <1초

L1: 방향 제시 ("어떤 부분에서 막혔어?")
  ↓ 학생 답변
L2: 핵심 파악 ("이 문제에서 구하는 값이 뭐야?")
  ↓ 학생 답변
L3: 구체 단서 ("삼각형 넓이 공식 중에 두 변과 끼인각을 쓰는 건?")
  ↓ 학생 답변
L4: 거의 정답 ("맞아! 그럼 여기서 a, b, C에 해당하는 값은?")
  ↓ 정답 입력 → L5에서 해설 공개
  ↓ 또는 L4까지 실패 → L5에서 전체 해설
```

**핵심**: 정답을 바로 주지 않고 **사고 유도**.

### 6.2 풀림 해설 12-섹션 심화 탐색

[03-features-and-ia.md 4.5장](03-features-and-ia.md) 참조. 학생은 12 섹션을 **순차 또는 비순차**로 탐색 가능.

### 6.3 라이브러리 미디어 생성 → 학습 연결

```
[라이브러리 만들기]
  ├─ 이미지: "사인법칙 3가지 증명" → 화풍(다이어그램) + 비율(16:9) → 생성
  ├─ 쇼츠: "조건부확률 핵심 30초" → 강사(여자 밝은톤) + 자막 ON → 생성
  ├─ 오디오: "삼각함수 개념 암기" → 목소리(남자 차분) + 속도(1.0×) → 생성
  └─ 카드: "cos vs sin 비교" → 스타일(비교표) + 색상(파랑) → 생성
  ↓ 1.8초 toast: "내 자료실에 저장했어요"
  ↓
[내 자료실] 관리
  ├─ 그리드 또는 목록 뷰 토글
  ├─ 종류·소유자·정렬 필터
  └─ 카드 클릭 → 상세 (prompt 공개 + 비슷하게 다시 만들기)
```

### 6.4 인터랙션 우선순위

- **학습 중단 없는 흐름** 우선: 모달보다 **사이드 패널/인라인** 사용
- **타이머 항상 가시** (블록 실행 중)
- **AI 응답은 스트리밍** (T3는 필수, T2도 권장)
- **에러 상태**: 재시도 버튼 + 상세 메시지 (학생이 무엇을 다시 시도해야 하는지 명확)
- **버튼 어포던스**: 주요 액션 버튼은 [08-design-system.md § 7.3](08-design-system.md)을 따른다. 태그·뱃지 모양으로 액션을 숨기지 않는다.

### 6.5 도메인 이동 알럿 정책 (Cross-Domain Transition Alert)

**원칙**: 화면 흐름은 예측 가능해야 한다. 학습자(고등학생)가 같은 도메인 안에서 작업을 마칠 수 있도록 설계하고, **부득이하게 도메인을 옮겨야 할 때만** 사용자에게 사전 안내한다.

#### 6.5.1 도메인 단위 정의

알럿이 발동되는 "도메인"은 **풀림 6 top-level 도메인**만:
- 풀림 플래너 (`/planner/*`)
- 풀림 Q (`/q/*`)
- 풀림 클래스봇 (`/classbot/*`, `/teacher/classbot/*`)
- 풀림 라이브러리 (`/library/*`)
- 풀림 스튜디오 (coming-soon)
- 풀림 스토어 (coming-soon)

> Q 내부 4 서브도메인 (`infinity`, `talk`, `analysis`, `review`) 간 이동은 **알럿 발동 대상 아님**. 학생 핵심 6단계 루프(진단 → 풀이 → 해설 → 정복 → 갱신)가 Q 안에서 자연스럽게 흐르도록 보장.

#### 6.5.2 알럿 트리거 조건

| 이동 유형 | 알럿 발동 |
|---|---|
| 사용자가 **글로벌 nav · 사이드바 · 도메인 메뉴**를 직접 클릭 | ❌ 발동 안 함 |
| 사용자가 **명시적 도메인 메뉴 이동** | ❌ 발동 안 함 |
| **시스템 자동 이동 / 추천 링크 / 액션 결과 이동** | ✅ 발동 |
| **딥링크로 다른 도메인 진입** (블록 자동 진입 등) | ✅ 발동 |
| **버튼 액션이 부수효과로 도메인 변경** (예: 정복 완료 → 플래너 자동 갱신) | ✅ 발동 |

> 사용자가 의도적으로 nav를 누르는 건 면제 — 자기 의도라서. 알럿이 필요한 건 **시스템이 능동적으로 학생을 다른 도메인으로 이끌 때**.

#### 6.5.3 알럿 형태 (확인형 모달)

- **타입**: Confirm modal (차단형)
- **취소 시**: 현재 도메인에 머무름
- **확인 시**: 목적 도메인으로 이동
- **콘텐츠**:
  - 제목: "다른 도메인으로 이동합니다"
  - 본문: "지금 [현재 도메인]에서 [목적 도메인]으로 이동합니다. 계속하시겠어요?"
  - 변경되는 컨텍스트 요약 (예: "오답 카드 2개를 [풀림 플래너]에 추가하고 일정을 갱신합니다")
- **CTA**: `[취소]` (보조), `[이동]` (Primary)
- **선호 기억**: "다음부터 자동으로 이동" 토글 — 사용자가 켜면 같은 패턴은 알럿 생략 (사용자 단위 저장)

#### 6.5.4 적용 예시 (기존 흐름 → 신규 알럿)

| 흐름 | 발동 여부 | 알럿 내용 |
|---|---|---|
| UC-1: 정복 완료 → 플래너 자동 갱신 ([04-ux-flow.md UC-1 5번](04-ux-flow.md)) | ✅ | "오답 정복 결과가 [플래너]에 반영됩니다. 이동해서 일정을 확인할까요?" |
| UC-2: 플래너 NEXT BLOCK [지금 시작하기] → Q 딥링크 | ✅ | "[풀림 Q] 무한풀기로 이동해 블록을 시작합니다." |
| UC-3: 모의고사 분석 → "취약 단원 2주 집중" 자동 플래너 블록 생성 | ✅ | "[플래너]에 취약 단원 보강 블록을 자동 생성합니다." |
| UC-4: 라이브러리 → [Q에서 풀어보기] 버튼 | ✅ | "[풀림 Q]에서 이 주제로 문제를 풀어볼 수 있어요." |
| 글로벌 nav `Q → 플래너` 클릭 | ❌ | 면제 (사용자 명시 액션) |
| Q 내부 `solve → talk` 이동 | ❌ | 면제 (서브도메인 간) |

#### 6.5.5 구현 참고

- 알럿 컴포넌트는 디자인 시스템 모달 베이스 사용 ([08-design-system.md](08-design-system.md) 4.1 참조)
- 알럿 결정 로그를 분석에 활용 (어떤 cross-domain 이동이 자주 거부되는지 → 흐름 재설계 신호)
- 시스템 자동 이동 직전에 발동, 백그라운드 부수효과(데이터 저장 등)가 이미 일어났다면 **롤백 가능 여부 명시**

### 6.6 오버플로 처리 규칙 (Overflow Handling)

**원칙**: 콘텐츠 길이는 가변이지만 학습 흐름은 끊기지 않아야 한다. 헤더(타이틀·맥락)와 푸터(CTA·확인/취소)는 항상 가시 영역에 남아야 하며, 본문이 viewport를 초과하면 **본문 영역만 스크롤**로 처리한다. 헤더/푸터가 viewport 밖으로 잘려 사용자가 다음 행동을 결정하지 못하는 상태는 버그로 간주한다.

#### 6.6.1 컨테이너 분류와 처리

| 컨테이너 유형 | 위치 정책 | 오버플로 처리 |
|---|---|---|
| **다이얼로그·모달** | viewport 중앙 | `max-height: calc(100dvh - 2rem)` + 본문만 스크롤, 헤더·푸터 sticky |
| **시트(Sheet/Drawer)** | viewport 엣지 부착 | `height: 100dvh` + 본문만 스크롤, 헤더·푸터 sticky |
| **팝오버·드롭다운·콤보박스** | 트리거 앵커 | 자체 collision boundary + `max-height` 내부 스크롤 (Base UI 기본 동작 유지) |
| **카드·섹션 (페이지 흐름 안)** | 페이지 인라인 | **내부 스크롤 만들지 않음** — 페이지 스크롤에 위임 (중첩 스크롤 금지) |
| **전용 풀스크린 영역 (OMR·시험 진행)** | 페이지 전체 | 페이지 스크롤 또는 영역별 독립 스크롤. 헤더·타이머는 sticky |

#### 6.6.2 베이스라인

- **단위는 `dvh`**: 모바일 주소창 동적 변화에 대응. `vh`/`%` 사용 금지.
- **viewport 여백 `2rem` 보존**: 다이얼로그가 화면 끝에 붙지 않도록.
- **헤더·푸터 sticky**: `top-0` / `bottom-0` + 본문 `overflow-y-auto`. 스크롤 가능성을 시사하는 시각 단서(스크롤바 또는 fade mask)를 노출한다.
- **중첩 스크롤 금지**: 한 화면에 두 개 이상의 독립 스크롤 영역을 만들지 않는다. 단, 페이지 스크롤과 직교하는 좌·우 분할 패널은 예외.
- **무한 스크롤은 별개 패턴**: 본 규칙은 "유한 콘텐츠가 viewport를 초과하는 경우"에 적용. 무한 스크롤(피드·타임라인)은 별도 명세 대상.

#### 6.6.3 적용 예시

| 케이스 | 처리 | 상태 |
|---|---|---|
| 시험 모드 다이얼로그 (`/q/infinity/solve`) — 시험 세트 N개 + 차단/활성 기능 grid + 룰 박스 + 푸터 | 다이얼로그 `max-h-[calc(100dvh-2rem)]`, 시험 세트 리스트만 스크롤, 헤더(제목)·푸터(취소/시작) sticky | ✅ 베이스 처리 ([dialog.tsx](../../src/components/ui/dialog.tsx)) · 검증 캡처 [output/live-shots/overflow-fix/dialog-compact-final.png](../../output/live-shots/overflow-fix/dialog-compact-final.png) · 후속 cosmetic [2026-05-07 dialog footer cleanup](../archive/2026-05-07_dialog-footer-bg-cleanup.md) (`bg-muted/50` → `bg-muted` 불투명화) |
| 풀이 세션 선택 드로어 | Sheet `h-[100dvh]` (left/right), `max-h-[100dvh]` (top/bottom), 세션 카드 리스트만 스크롤, 닫기 버튼 sticky | ✅ 베이스 처리 ([sheet.tsx](../../src/components/ui/sheet.tsx)) |
| 드롭다운 메뉴 (프로필·역할 전환 등) | `max-h-(--available-height)` Base UI collision-aware, 자체 스크롤 | ✅ 베이스 기본 ([dropdown-menu.tsx](../../src/components/ui/dropdown-menu.tsx)) |
| 코치 채팅 패널 | 메시지 리스트만 스크롤, 입력창 footer sticky | (인라인 패턴 — 베이스 컴포넌트 외) |
| 풀이 워크스페이스의 문제 카드 N개 | 페이지 스크롤에 위임 (카드 내부 스크롤 만들지 않음) | (페이지 패턴) |

#### 6.6.4 검증

- **최소 viewport 통과 기준**: `375×667` (Compact 최소), `1280×720` (Comfortable 최소). 콘텐츠 항목 수 N을 `availableExams` / 검색 결과 / 카드 리스트 등 실제 데이터의 최댓값까지 늘려도 다음을 만족해야 한다.
  - 헤더(타이틀)에 즉시 도달 가능
  - 푸터(CTA)에 스크롤 없이 또는 한 번의 스크롤로 도달 가능
  - 가로 스크롤 미발생
- **베이스 컴포넌트 책임**: `components/ui/{dialog,sheet,popover,dropdown-menu,select}.tsx`는 본 규칙을 기본 동작으로 충족해야 한다. 페이지·도메인 컴포넌트가 매번 `max-h`를 덧붙이는 구조는 회귀 위험이 큼 — 베이스에서 처리.
- **다이얼로그 footer는 sticky + opaque (`bg-muted`)**: 본문 스크롤 시 footer 뒤로 콘텐츠가 비치지 않도록 불투명 배경 유지 ([2026-05-07 dialog footer cleanup plan](../archive/2026-05-07_dialog-footer-bg-cleanup.md) 처리 완료).

## 9. 클래스봇 UX 패턴 (Classbot-specific)

> 출처: `input/design-system/private-classbot.md` (Playwright 사적 감사 2026-05-19). [08-design-system.md § 15](08-design-system.md)의 토큰·컴포넌트와 짝을 이루는 **인터랙션/플로우 명세**.

### 9.1 홈 4블록 우선순위 (시간 가치순)

학생 `/classbot` 홈은 **"지금 일어나는 일"** 부터 노출한다. 회귀 사례: LIVE 진행 중 블록이 3번째에 있어 학생이 라이브를 놓침.

| 우선 | 블록 | 노출 조건 | 비주얼 |
|---|---|---|---|
| 1 | **지금 LIVE** | 있을 때만 | navy solid + lime 4px 좌측 라이너 + M5 펄스 + "들어가기" CTA |
| 2 | **내 클래스봇** (5개) | 항상 | 봇별 시그니처 컬러 점 + 메시지/과제 미리보기 포함 (§ 9.2) |
| 3 | **진행 중 과제 1개** (가장 임박) | 미완료 있을 때 | § 15.6 상태 컬러 매핑 적용 |
| 4 | **오늘의 KPI / 웰빙 한 줄** | 항상 | brand.50 카드 |
| 5 | **최근 리플레이** | 있을 때만 | 16:9 썸네일 리스트 (§ 9.10) |

### 9.2 봇 카드 정보 보강

```
┌──────────────────────────┐
│ [🧑‍🏫] 수학이 형 ●LIVE     │  ← 봇 시그니처 컬러 점 + LIVE 펄스
│       김수학 · 고2 미적A │
│  ─────────────────────── │
│  💬 "오늘 진도 잘 따라왔어"  ← 마지막 메시지 1줄 (text.tertiary)
│  📋 미완료 1 · 라이브 1   │  ← 미완료/라이브 카운트
│                  [입장 →] │  ← 1차 CTA만 (LIVE면 "입장", 아니면 "대화")
└──────────────────────────┘
```

- 마지막 메시지 1줄 + 미완료/라이브 카운트 + 1차 CTA. iMessage 채팅 리스트 패턴.
- 데스크탑: 5봇 → **5열 단일 행**(가로 카드, 비주얼 균형). 6봇 이상 → 첫 행 5 + `+ N개 더` 카드.
- 모바일: 1열 세로 — LIVE 봇 2개가 위 (회귀 사례: 분홍 틴트만으로는 흑백 모드 구분 안 됨 → 시그니처 점 + LIVE 펄스로 강화).

### 9.3 봇 메타 카드 (헤더) collapse

봇 채팅 화면 헤더. 회귀 사례: 모바일 388px 점유로 첫 메시지 1개만 보임.

| 상태 | 높이 | 노출 정보 | 트리거 |
|---|---|---|---|
| **collapsed (기본)** | 56px | `← [🧑‍🏫] 수학이 형 · 개념 모드(L3)        ⓘ ⌃` | 진입 기본 상태 |
| **expanded** | 180px | + 학원명 / 톤(친근체) / 응답시간(~3초) / 자동차단 안내 + 범위 변경 진입 | `⌃` 클릭 / 메타카드 영역 탭 |

룰:
- 모바일에서 **키보드 열림 시 자동 collapse** (§ 9.5).
- expanded 상태에서 스크롤 내리면 자동 collapse.
- 토큰·비주얼은 [08 § 15.5](08-design-system.md).

### 9.4 봇 스위처 칩 (탭+드로어 하이브리드)

```
┌─────────────────────────────────────┐
│ [수학이 형 ●LIVE]  [영어 누나]  …    │  ← horizontal scroll 5개
│                                  [≡] │  ← drawer: 검색·즐겨찾기
└─────────────────────────────────────┘
```

- 가로 스크롤 5칩 + 오른쪽 끝 `[≡]` drawer.
- 활성 봇: **봇 시그니처 컬러 배경 + 흰 글자**.
- LIVE 봇: 좌측 빨간 펄스 점 (M5). 대기 봇: 회색 정적 점.
- 봇 전환 시 M6 모션 (cross-fade 160ms + 메타카드 slide-down + 라이너 swipe).

### 9.5 **모바일 키보드 입력바 sticky (Critical)**

P0 회귀: 모바일 viewport 460px 시뮬레이션에서 입력바·빠른칩 사라짐. 봇 헤더와 첫 메시지만 보임. **`visualViewport` API 미적용** 추정.

#### 9.5.1 요구사항

1. **입력바 + 빠른칩**은 `visualViewport.height`를 기준으로 키보드 위에 sticky 고정.
2. 키보드 열림 시 **메시지 영역 자동 스크롤** — 마지막 메시지가 입력창 바로 위에 위치.
3. **봇 메타 카드 자동 collapse** (§ 9.3) — 56px만 차지하도록.
4. iOS Safari 주소창 동적 변화 대응 — `100dvh` 사용 ([§ 6.6.2](#)와 정합).

#### 9.5.2 구현 가이드

```ts
// useVisualViewport (개념)
const vv = window.visualViewport;
const keyboardOpen = vv && vv.height < window.innerHeight - 100;
// inputBar.style.bottom = `${window.innerHeight - vv.height - vv.offsetTop}px`
```

- `vv.addEventListener('resize', …)`·`'scroll'`로 추적
- iOS Safari·Android Chrome 모두 테스트 (양쪽 동작 다름)

#### 9.5.3 검증

- 모바일 viewport 375×667 / 390×844에서 키보드 열림 → 입력바·전송 버튼·빠른칩 모두 가시
- 마지막 메시지가 입력바 위에서 잘리지 않음
- 메타카드 자동 collapse 확인

### 9.6 빠른 칩 (Quick Reply) 동적 추천

회귀 사례: 4개 하드코딩 (`극값 어떻게 찾아요? / 오늘 수업 요약해줘 / …`). 봇·과목 무관.

#### 9.6.1 추천 룰 3종

| 시나리오 | 추천 소스 | 예시 |
|---|---|---|
| **대화 시작** (메시지 0건) | 봇·과목 입문 질문 | "오늘 진도 알려줘" / "지난 시험 분석해줘" |
| **마지막 봇 발화 기반** | 봇 응답의 키워드 + 후속 질문 | 봇이 "극값" 설명 → 칩 "극값 예제 더 보여줘" |
| **시간/상황 기반** | 시간대 + 웰빙 신호 | 새벽 진입 → "오늘 컨디션 안 좋아" 같은 웰빙 진입 칩 |

#### 9.6.2 표시

- 인풋 위에 **2줄 stack** (1줄 3칩 × 2 = 최대 6칩)
- 칩 좌측에 봇 시그니처 컬러 2px 라이너 (봇 추천임을 시각화)
- 등장 모션: M7 stagger (60ms 시차)

### 9.7 입력바 첨부·음성

```
┌─────────────────────────────────────────────────────┐
│ [+] [🎤] [수학이 형에게 물어보세요…           ] [→] │
└─────────────────────────────────────────────────────┘
```

| 진입점 | 동작 |
|---|---|
| `[+]` 탭 | **drawer 위로** — 사진 / 카메라 / 필기 / 문제 캡처 / 음성 메모 |
| `[🎤]` 길게 누름 | 음성 입력 — 영어 누나 발음 입력에 필수 |
| 입력 영역 | multiline 자동 확장 (max 5 lines) — 이후 자체 스크롤 |
| 전송 `[→]` | brand.600 원형, M1 모션으로 메시지 mount |

- placeholder: 봇 이름 + 시그니처 prompt 예시 (예: 수학이 형 — "막힌 문제 사진 찍어 봐도 돼")
- Scope·Tone 표시는 **입력바 풋터에 두지 않음** ([07 § 5.3](07-branding.md), [08 § 15.7](08-design-system.md)).

### 9.8 채팅 메시지 그루핑 / 시간 디바이더

| 패턴 | 룰 |
|---|---|
| **연속 발화** | 같은 봇 3분 이내 메시지는 아바타·이름 생략 + 32px 들여쓰기 |
| **시간 디바이더** | 날짜 변경 시 `─── 오늘, 5월 19일 ───` 가운데 정렬. 12px / text.tertiary / 32px 상하 여백 |
| **메시지 타입** | 6종 (text / problem-card / explain-step / reference-link / image / audio) — [08 § 15.1.3](08-design-system.md) |

- 호명("서연") 유지 — OK 패턴
- 출력 전 M9 (응답 wave bar) — 봇이 "지금 말 시작" 시각 cue

### 9.10 LIVE 수업 + 리플레이

#### 9.10.1 LIVE 카드 시그니처

- navy solid + 4px lime 좌측 라이너 + LIVE 뱃지 M5 펄스 (자세히는 [08 § 15.3](08-design-system.md))
- **시청자 수 라이브 카운터**: 숫자 변동 시 spring 180ms, +5 floating

#### 9.10.2 리플레이 카드 위계

| 우선 | 카드 | 비주얼 |
|---|---|---|
| 1 (이어 보기) | navy + lime progress + 큰 CTA | **1.5배 크기**, hover 시 lime glow |
| 2 (오늘 끝남) | navy solid + 정답률/내 활동 stats | 보통 크기 |
| 3 (전체 수업) | white card + 16:9 썸네일 | 리스트형 |

#### 9.10.3 16:9 자동 캡처 썸네일

- 수업 핵심 슬라이드/판서 첫 프레임을 자동 캡처해 썸네일로 사용
- 카드 좌측 16:9, 우측에 정답률·내 활동 stats
- 정적 placeholder 금지 (회귀 사례: 다크 그라데이션만으로는 식별 불가)

### 9.11 온보딩 (소개하기) — 인터랙티브 데모 + sticky 진척 바

#### 9.11.1 sticky 진척 바

- 페이지 상단 sticky: `4/7 ●●●●○○○ 4분` — 끝까지 스크롤 동기 부여
- 단계 카드 좌측 큰 숫자 ● 유지 (일관성)

#### 9.11.2 단계 3 (봇 채팅 소개) 인터랙티브 데모

7단계 중 **봇 채팅 단계만** 실제 데모 진입 가능:
- 데모 입력창 (API 안 연결, 시연만)
- 학생이 "안녕!" 치면 봇이 "안녕 서연! 나는 …" 1회 응답 (첫 손맛)
- 데모 종료 후 다음 단계 진행

#### 9.11.3 끝 CTA

마지막 단계 후 dead-end 금지:
- 큰 CTA "수학이 형과 첫 대화 시작 →" ([07 § 4.10.1](07-branding.md))

### 9.12 브레드크럼 자동 dedupe

회귀 사례: "풀림 클래스봇 > 풀림 클래스봇" 같은 동일 라벨 연속 표시.

- 같은 라벨이 연속해서 나타나면 **뒤쪽을 자동 제거**.
- 구현 위치: `components/shell/breadcrumb.tsx`의 트레일 빌드 단계에서 `Array.filter`로 인접 중복 제거.
- 모든 도메인에 적용 (클래스봇만의 룰 아님 — shell 전역).
