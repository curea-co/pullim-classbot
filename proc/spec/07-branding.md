## 2. Brand Personality

### 2.1 핵심 성격 (5가지)

| 성격 | 의미 | 반대 (피해야 할 것) |
|------|------|------------------|
| **친근한** | 학생 옆에 있는 형·누나 같은 톤 | 차갑고 사무적 |
| **전문적** | 교육과정·IRT 등 데이터 기반 신뢰 | 어설픈 일반론 |
| **격려하는** | "할 수 있어요" 메시지 | 강압적·꾸짖음 |
| **명확한** | 핵심을 3초 안에 전달 | 장황·중언부언 |
| **혁신적** | AI 기반 차별점이 드러남 | 기존 인강의 디지털화 |

### 2.2 페르소나로서의 풀림
- **나이**: 20대 후반 형/누나 (학생 입장에서 살짝 윗사람)
- **성격**: 똑똑하지만 뽐내지 않음, 학생 편에서 같이 고민
- **말투**: 친근한 존댓말, 가끔 이모지 (과하지 않게)

## 3. Tone & Voice

### 3.1 톤 (Tone) — 상황별

| 상황 | 톤 | 예시 |
|------|---|------|
| 일반 안내 | 친근·중립 | "오늘 어떤 문제 풀어볼까요?" |
| 격려 | 따뜻·긍정 | "조금만 더 가면 돼요. 잘하고 있어요." |
| 정답 / 성공 | 축하 | "완벽해요! 🎉 다음 문제 가볼까요?" |
| 오답 | 부드럽지만 명확 | "이번엔 아쉽네요. 어디서 막혔는지 같이 봐요." |
| 시스템 오류 | 신뢰·해결책 제시 | "잠시 문제가 생겼어요. 잠시 후 다시 시도해주세요." |
| AI 제한 (Scope L1) | 정중·이유 명시 | "지금은 시험 시간이라 도와드릴 수 없어요. 끝나면 같이 풀어봐요." |

### 3.2 보이스 (Voice) — 절대 원칙

✅ **DO**
- 친근한 존댓말 ("~해요", "~볼까요?", "~돼요")
- 학생 입장에서 같이 고민하는 표현 ("같이", "함께", "우리")
- 짧고 명확한 문장 (한 문장 ≤ 25자 권장)
- 학생을 응원하는 톤 ("잘하고 있어요", "한 번 더 해봐요")

❌ **DON'T**
- 반말 ("야", "~해라", "~다")
- 과한 격식 ("~하시기 바랍니다", "~해주십시오")
- 학생을 가르치려 드는 톤 ("이건 이렇게 해야 해요", "당연히 알아야 해요")
- 부정적 평가 ("이걸 모른다고?", "쉬운 건데")
- 영문 코드명 노출 ("Infinity Engine", "Socrates", "Conqueror") — § 5.1
- 어려운 한자어 (잔존·임박·차시·진척도 등) — § 6 교체 표 적용
- 학술 기호·약어 (θ, σ, IRT 등) — 학생 UI에선 한국어로 풀어 쓴다 (§ 5.2)

### 3.3 학생 vs 교사 UI의 톤 차이

| 영역 | 톤 |
|------|----|
| 학생 UI | 친근한 존댓말 |
| 교사 UI | 정중한 존댓말, 약간 더 격식 ("~해주세요", "~하실 수 있어요") |
| 학부모 UI | 정중·신뢰 ("~하시면 됩니다", "자녀분") |
| 시스템 알림 | 중립적 ("~되었습니다", "~필요해요") |

## 6. UX Writing — 한자어 · 표현 절감

> **권위**: [proc/research/2026-05-06_hanja-vocabulary-ux-policy.md](../research/2026-05-06_hanja-vocabulary-ux-policy.md)
> 배경(OECD PIAAC, 한국형 실질적 문맹 분석, UI vs 콘텐츠 분리 원칙)은 권위 문서 참조. 본 절은 적용 규칙만.

### 6.1 핵심 원칙 — UI 카피 vs 학습 콘텐츠 분리

| 영역 | 한자어 정책 |
|---|---|
| **UI 카피** (메뉴·버튼·빈 상태·에러·온보딩·툴팁·알림·차트 라벨) | **최소화** — 고등학생이 한 번 읽고 의미가 잡혀야 함 |
| **학습 콘텐츠** (문항·해설·개념 설명·교과서 인용) | **유지** — 시험 환경 정합 우선. 임의 평탄화 금지 |

> 경계선이 흐려지면 "쉬워서 좋다"가 아니라 **"공부에 도움 안 되는 앱"**이 된다. 풀림 도메인명("플래너 / Q / 클래스봇 / 라이브러리")이 한자어 도메인명("학습관리·문제은행")을 의도적으로 피한 결정 자체가 본 원칙의 적용 사례.

### 6.2 한자어 교체 표 (UI 카피)

| 한자어 | 교체안 | 적용 영역 예 |
|---|---|---|
| 잔존 / 잔존률 | 남은 기억 / 기억 정도 | 망각 곡선·복습 카드 |
| 잔여 학습량 | 남은 공부 / 남은 학습 | 진도 카드 |
| 임박 / 임박한 | 곧 / 곧 다가오는 | 일정·알림 |
| 차시 | 회차 | 강의·콘텐츠 카드 |
| 미달 / 미충족 | 부족 / 아직 모자람 | 점수·달성률 |
| 진척도 | 진행률 | 대시보드 |
| 고지 / 공지 | 알림 | 시스템 |
| 회신 | 답장 | 메시지 |
| 익월 / 익일 | 다음 달 / 다음 날 | 일정 |
| 상기 / 하기 | 위 / 아래 | 안내 |
| 종료 | 끝남 / 마침 | 상태 |
| 본인 인증 | 본인 확인 / 내 정보 확인 | 인증 |

> 표는 살아있는 문서 — 회귀 사례 발견 시 행 추가.

### 6.3 불필요한 텍스트·컴포넌트 절감

작성·리뷰 시 의심 항목:
- **장식형 부제**: 본문이 이미 말한 내용을 다시 말하는 부제 → 삭제
- **과잉 배지/태그**: 한 카드에 다른 의미 배지가 3개 이상이면 정보 위계 재설계
- **전문가 톤 압축 카피**: "θ 0.42 → +0.05 변동, σ 0.08" 같은 한 줄 다지표 압축은 분리 또는 요약 문장으로 변환
- **의사결정 무관 메타**: 학생 행동 선택에 영향 없는 메타데이터는 상세 페이지로 이동

### 6.4 콘텐츠 영역 한자어 — 부가가치로 전환 (향후)

학습 콘텐츠(문항·해설)에서 한자어를 빼지 않는 대신, 모르는 한자어를 **학습 자산**으로 만드는 기능을 별도 plan으로 검토:
- 한자어 long-press → 한자 + 음 + 뜻 툴팁
- 어휘 메모리 — 풀림 Q의 메모리 도메인과 결합
- 단어 형태소 분해 — "잔존(殘存)" → 殘(남을 잔) + 存(있을 존)

상세 검토는 [research 문서 § 2.3](../research/2026-05-06_hanja-vocabulary-ux-policy.md) 참조. 본 spec에서는 "콘텐츠는 평탄화하지 않는다"만 강제.

### 6.5 검증 질문

신규/리뷰 화면 카피마다:

- [ ] § 5.2 학술 기호·약어를 한국어로 풀었는가
- [ ] 같은 정보를 두 번 말하거나, 의사결정 무관 메타가 쌓이지 않는가
