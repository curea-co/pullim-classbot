## 1. AI 명령지침 (이 spec 한정)

- 이 문서를 단일 진실원으로 본다. 권위 문서와 충돌하면 권위 문서를 따르고 본 spec을 갱신한다.
- **개발 시작 전 반드시 검토할 것**:
  1. `gradingQueue` mock의 모든 필드를 화면에서 소비하는지
  2. "AI 초안 = 제안, 교사 = 판결" 디자인 원칙 (핸드오프 § 2.5) 위반하지 않는지
  3. 객관식·서술형·단답형 3타입 모두 동일 큐에서 처리되는지
- 추출본 정책: 다른 도메인(플래너/Q/라이브러리) 의존 금지. 채점 허브는 클래스봇 단일 도메인 내부에서 자급한다.
- **금지**: AI 점수를 학생/학부모에게 직접 노출. 반드시 교사 승인 후에만 학생·학부모 가시.

## 2. 제품 정의 (Problem · Goal · Persona)

### 2.1 Problem Statement
공교육 교사·학원 강사의 서술형 채점은 1인당 2주 → 평균 채점 소요시간이 학생 피드백 사이클을 끊는다. 객관식은 자동화됐지만 서술형·단답형은 여전히 수작업이고, 채점 일관성도 교사 컨디션에 좌우된다.

### 2.2 Product Goal
- **G1**: 서술형 채점 소요시간 **80% 단축** (KPI § 10.2)
- **G2**: 교사는 "초안 검수자"로 역할 전환 — 점수 작성 → 점수 확인·수정
- **G3**: 교사 수정 데이터가 루브릭 재학습으로 환류 (선순환 폐쇄)

### 2.3 Persona별 영향

| 페르소나 | 역할 | 이 기능에서 무엇이 달라지나 |
|---|---|---|
| **P1 교사** (주) | 채점자 | 큐에 쌓인 AI 초안을 검수·승인. 변경률 20% 초과 시 루브릭 재학습 제안 수락 |
| **P2 학생** | 응시자 | 제출 후 평균 24시간 → 6시간 내 피드백 수신 |
| **P3 학부모** | 열람자 | 최종 점수만 노출 (AI 초안은 안 보임) |
| **P4 학원 관리자** | 감독자 | 강사별 검수 평균 소요 / 변경률 KPI 열람 |

## 3. 핵심 기능 정의 (FSD)

### 3.1 MoSCoW

#### Must (MVP)
- [M1] 채점 큐 화면 — 상태별 필터 (대기/검토중/완료)
- [M2] 채점 상세 화면 — AI 초안 + 루브릭 항목별 점수 표시
- [M3] 교사 인라인 편집 — 점수·코멘트·루브릭 항목 수정
- [M4] 승인 / 오버라이드 액션 — 학부모 가시화 트리거
- [M5] 변경률 자동 계산 + 20% 임계 알림

#### Should (v1)
- [S1] 상위 10% / 하위 10% 자동 분석 패널 (T3)
- [S2] 일괄 작업 — 다중 선택 → 일괄 승인
- [S3] 루브릭 재학습 제안 모달 — 교사 승인 시 다음 과제부터 적용
- [S4] 학생별 채점 이력 사이드 패널

#### Could (v2)
- [C1] 손글씨 답안 OCR + 채점 (수학 풀이 과정)
- [C2] AI 음성 코멘트 자동 생성 (교사 목소리 TTS)
- [C3] 표절·AI 작성 의심 플래그

#### Won't (현 단계 제외)
- 자동 점수 학부모 직접 발송 (반드시 교사 승인 게이트 통과)
- AI가 만점 처리 (모든 항목 최소 1초간 교사 시선 보장 — 안티 패턴 방지)

### 3.2 Sitemap 변경

```
/teacher
└─ /teacher/grading                 ← 신규 · 채점 큐 (필터·정렬·일괄 액션)
   ├─ /teacher/grading/[id]         ← 신규 · 채점 상세 (AI 초안 검수)
   └─ /teacher/grading?filter=…     ← URL 상태 (status, type, bot)
```

학생 영역에 채점 진입 없음 — 학생은 "결과 보기" CTA로 [학생 과제 풀이 결과 페이지](12-student-assignment-solve.md § 3.2)에서 최종 점수만 확인.

### 3.3 Screen Spec

#### 3.3.1 채점 큐 (`/teacher/grading`)
| 구성 | 설명 |
|---|---|
| **상단 KPI 바** | 대기 N건 · 검토중 N건 · 오늘 승인 N건 · 평균 변경률 N% |
| 필터 칩 | 상태(`queue/reviewing/approved/overridden`), 타입(`essay/short/numeric`), 봇, 과제 |
| 정렬 | 제출 시간 / AI 신뢰도 / 위기 신호 우선 |
| **채점 행** | 학생명·과제·타입·AI 점수·신뢰도·상태·"검수" CTA |
| 위기 신호 인디케이터 | `aiConfidence < 70` 또는 응답 빈약·감정 키워드 |
| 일괄 액션 바 | 다중 선택 시 노출 — "선택 N건 승인" / "일괄 오버라이드" |
| 사이드 패널 (lg+) | 오답 클러스터 TOP 3 (이 과제 한정) — 패턴별 학생 수 |

#### 3.3.2 채점 상세 (`/teacher/grading/[id]`)
| 구성 | 설명 |
|---|---|
| 헤더 | 학생명·과제명·제출 시각·"이전/다음 학생" 네비 |
| 좌측 — 문제 + 학생 응답 | KaTeX 렌더 · 학생 원본 텍스트 · OCR (있을 시) |
| 중앙 — 루브릭 카드 | 항목별 (criterion · weight · score · reason) · 인라인 슬라이더로 점수 수정 |
| 우측 — AI 초안 코멘트 + Tier 표시 | 교사 편집 가능 textarea · 변경 시 변경률 실시간 표시 |
| **변경률 패널** | 점수·코멘트 변경 % · 20% 초과 시 "루브릭 재학습 제안" 활성 |
| 액션 바 | `[승인]` · `[수정 후 승인]` · `[전체 재채점 요청]` · `[1:1 면담 메모]` |
| 사이드 — 이 학생 최근 5회 | 과거 채점 이력 추세 (점수·코멘트 유형) |
| 위기 게이트 | 응답에 자살·우울 키워드 발견 시 자동 경고 모달 (3자 알림 트리거) |

## 4. UX Flow

### 4.1 Flow G1 — 교사 일상 채점 (가장 빈번)
```
1. 교사 홈 → "서술형 채점 대기 12건" 카드 클릭
2. /teacher/grading 진입 → 큐 정렬: AI 신뢰도 낮은 순
3. 첫 번째 학생 행 클릭 → 상세 진입
4. 루브릭 점수 1개 수정 (-2) + 코멘트 한 줄 추가
5. [수정 후 승인] → 다음 학생 자동 이동
6. 일관 패턴: 5명 채점 = 약 7분 (수동 30분 → 80% 단축)
```

### 4.2 Flow G2 — 루브릭 재학습 환류
```
1. 교사가 같은 항목("표기 정확성")을 5명 연속 -5점 추가 차감
2. 시스템: 변경률 누적 24% — 임계 초과
3. 상세 페이지 우측에 "루브릭 재학습 제안" 카드 점등
4. 교사 클릭 → 변경 패턴 요약 모달
5. [다음 과제부터 적용] → 루브릭 v2 활성, 이력 감사 로그 기록
```

### 4.3 Flow G3 — 위기 신호 발견 (응답 빈약)
```
1. 학생 도현 응답 = "잘 모르겠음. 미분이 너무 어렵다"
2. AI 초안: 8/20 · aiConfidence 58 · 코멘트에 "응답 부족"
3. 큐에서 위기 인디케이터 점등
4. 교사 진입 → "1:1 면담 메모" 클릭
5. 메모 작성 → 리포트 센터의 학생 개인 리포트에 자동 첨부
6. 도현이 다음 로그인 시 "선생님이 도현이를 응원해" 알림 (점수 노출 전)
```

### 4.4 RBAC

| 행위 | Owner | Manager | Assistant | Learner |
|---|---|---|---|---|
| 큐 열람 | ✅ 기관 전체 | ✅ 본인 봇 | ✅ 조회만 | ❌ |
| 점수 수정 | ✅ | ✅ | ❌ | ❌ |
| 승인 | ✅ | ✅ | ❌ | ❌ |
| 루브릭 재학습 | ✅ | ✅ | ❌ | ❌ |
| 최종 점수 열람 | — | — | ✅ | ✅ 본인만 |

## 5. 운영 로직 (Business Rules)

### 5.1 변경률 계산식
```
overrideDelta = |draftScore - finalScore| / maxScore × 100
```
- 누적 변경률 = 최근 30일 평균
- 30일 평균이 임계(20%) 초과 시 자동으로 루브릭 재학습 제안

### 5.2 상태 전이
```
queue ──교사 진입──> reviewing ──저장──> approved
                                  └──수정──> overridden  (overrideDelta 저장)
```
- `approved`와 `overridden` 모두 학부모·학생에게 가시화되는 종착 상태
- 재학습 후 같은 항목에 대해 점수가 자동 재계산되더라도 종착 상태는 보존

### 5.3 자동 처리 규칙
- `type === 'short' && aiConfidence ≥ 95` → T1 자동 채점 + 자동 승인 (교사 검수 스킵)
- `type === 'essay'` → 무조건 교사 검수 필수
- `type === 'numeric'` → 정답 일치 시 자동 승인, 부분 점수 케이스만 큐 진입

### 5.4 위기 게이트 (자동 차단)
응답 본문에 다음 정규식 매치 시 점수 표시 전에 경고 모달:
- 자살·자해 관련 키워드 (사전 정의 화이트리스트 기반)
- 감정 체크인 직전 3일 "힘듦" 누적 학생 응답

게이트 통과 후에만 점수 화면 노출. 동시에 [13. 리포트 + 감정 체크인](13-reports-and-emotion-checkin.md § 6.2) 의 3자 알림 트리거.

### 5.5 검증 규칙
| 항목 | 제약 |
|---|---|
| `rubric` 합 | 항상 100 (검증 실패 시 저장 차단) |
| 항목별 점수 | 0 ≤ score ≤ weight × 1.0 |
| 코멘트 길이 | 10 ≤ length ≤ 500자 |
| 승인 후 수정 | 금지 (수정 필요 시 별도 "재채점 요청" 로그) |

## 6. ERD 보강

## 8. 브랜딩 / UX Writing

### 8.1 톤
교사를 "판결자"가 아닌 "마지막 검수자"로 호칭. AI 초안은 항상 "초안" 또는 "제안"으로 표기 — 결코 "AI 채점 결과"라고 단독 표현 금지.

### 8.2 마이크로카피 테이블

| 상황 | 카피 |
|---|---|
| 큐 빈 상태 | "검수할 채점이 없어요. 학생들이 새로 제출하면 여기에 쌓여요." |
| AI 신뢰도 낮음 | "AI 신뢰도가 낮아요. 직접 보고 정하는 게 안전해요." |
| 변경률 20% 초과 | "루브릭이 학생 답과 어긋나는 것 같아요. 기준을 다시 학습할까요?" |
| 위기 게이트 발동 | "도현이 응답에 신경 쓸 표현이 있어요. 점수 전에 한 번 더 봐주세요." |
| 일괄 승인 확인 | "선택한 N건을 그대로 승인할게요. 되돌릴 수 없어요." |
| 재학습 제안 수락 | "다음 과제부터 새 기준이 적용돼요. 이전 점수는 그대로 남아요." |

### 8.3 금지어
- "AI가 채점했어요" → "AI 초안이에요"로 교체
- "정답" → 서술형은 "기준 응답" 또는 "모범 응답"
- "오답" → "보완 필요" (학생 가시 영역)
